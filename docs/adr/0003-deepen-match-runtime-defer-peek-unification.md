# Deepen the Match Runtime, Defer Full Peek Unification

The match runtime had drifted from ADR-0002's intent: instead of a deep rules module the scene advances through a small pure interface, `src/match/rules.ts` became a bag of ~30 functions over four state shapes (match, normal peeks, green chick, claim animations), and `MatchScene.update()` became the de-facto orchestrator — threading those states together every frame, manually passing peek state into the green-chick tick so it would avoid occupied spots, edge-detecting green-chick status flips to play sounds, and driving two separate claim paths.

We will restore ADR-0002 by making the match runtime a deep module: a stateful `Match` shell (the only thing scenes import) exposing `advance(deltaMs)`, `claim(spotIndex, player)`, and a semantic render-ready query/view. Both `advance` and `claim` return `MatchEvent[]`, so scenes have one reaction path for sounds and other side effects. The first required event vocabulary is intentionally small: normal chick claimed, green chick appeared, green chick claimed, and green chick missed. Match completion remains queryable state rather than an event in this pass.

The `Match` query/view should expose the scene-facing match facts — scores, remaining time, completion/winner, visible normal chick slots by hiding spot, and visible green chick by hiding spot/status — without exposing `PeekState`, `GreenChickState`, or Phaser-specific rendering details. Scenes still decide where a hiding spot is drawn, which Phaser sprite to move, and how tint/scale/physics are applied.

The module owns elapsed time, so the scene stops threading "now" into queries. Randomness is injected once when constructing the `Match`; internal pure helpers may still accept random values directly for focused deterministic tests. The gnarly decisions — spot allocation, refill timing, green-chick spawn-window math — stay as pure functions behind internal seams so they remain unit-testable without constructing a `Match`. Spot occupancy moves into an Arena that is the single source of truth (≤1 occupant per spot) and a passive allocator both internal machines call; this is the seam that removes the manual threading rather than relocating it. Arena may be exported from an internal module for focused unit tests, but it is not part of the scene-facing public API. The runtime lives in a small `src/match/` cluster with `match.ts` as the public surface.

We will NOT, in this pass, collapse normal chicks and the green chick into a single unified Peek instance driven by pluggable spawn policies. The Game Designer review is correct that this is the right long-term model and that it makes a third chick type or power-ups nearly free. But power-ups are coming "not soon," and with only two chick types the SpawnPolicy abstraction does not yet earn its keep ("one adapter = hypothetical seam, two = real"). Behind the unified `Match` interface, the two existing machines stay as they are, now owned by `Match` instead of threaded by the scene. Because the Arena seam and the public interface are built now, the later unification is a contained internal change that will not ripple to the scene or to the tests — those test observable behaviour through the public interface, not the internal shape. When a third chick type or power-ups actually land, revisit and unify then.

`ClaimAnimationState` stays a standalone pure helper this pass, driven by claim/advance events rather than treated as match state; relocating it into the view belongs with the render-list work (issue #22), not here. This pass also does not build the full render-list selector from issue #22: the match runtime exposes semantic visibility and occupancy facts, while Phaser sprite tint/scale/claim-pop rendering remains outside the runtime.
